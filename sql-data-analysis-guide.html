Intro

How to Use This Guide

Read it in order, but also copy each query into a SQL editor and run it against real tables. SQL is learned by reading results, not just memorizing syntax.

  • Each concept includes syntax, explanation, and a realistic data example.
  • Use the sample schema in this guide to practice queries against users, orders, transactions, and events.
  • Re-run examples with your own dataset once the logic makes sense.
  • Keep a notebook of query patterns; analysts reuse patterns constantly.
Goal: Build the exact SQL skill set expected of job-ready analysts in product, finance, marketing, operations, and strategy teams.
Notebook alternative: A runnable notebook version is available as sql-data-analysis-guide.ipynb for cell-by-cell practice.

Answer Key

Solved Practice Lab Queries

Use these after attempting the practice lab yourself. The point is to compare query structure, not just get the result.

1) Row Counts and Sanity Checks

SELECT COUNT(*) AS users_count FROM users;
SELECT COUNT(*) AS orders_count FROM orders;
SELECT COUNT(*) AS events_count FROM events;
SELECT COUNT(*) AS subscriptions_count FROM subscriptions;

Why: Confirms all source tables loaded correctly and lets you compare expected volume before analysis.

2) Distinct Country List

SELECT DISTINCT country
FROM users
ORDER BY country;

Why: Helps you understand segmentation fields and check for messy source values.

3) Revenue by Month and Country

SELECT
  DATE_TRUNC('month', o.order_date) AS month,
  u.country,
  COUNT(*) AS orders,
  SUM(o.quantity * o.unit_price - o.discount_amount) AS revenue
FROM orders o
JOIN users u
  ON o.user_id = u.user_id
WHERE o.order_status = 'completed'
GROUP BY 1, 2
ORDER BY 1, 2;

What it returns: One row per month-country combination, which is the standard shape for an executive revenue table.

4) Top Customers by Total Spend

WITH customer_spend AS (
  SELECT
    user_id,
    SUM(quantity * unit_price - discount_amount) AS total_spend
  FROM orders
  WHERE order_status = 'completed'
  GROUP BY user_id
)
SELECT
  user_id,
  total_spend,
  RANK() OVER (ORDER BY total_spend DESC) AS spend_rank
FROM customer_spend
ORDER BY spend_rank, user_id;

5) Monthly Running Revenue

WITH monthly AS (
  SELECT
    DATE_TRUNC('month', order_date) AS month,
    SUM(quantity * unit_price - discount_amount) AS revenue
  FROM orders
  WHERE order_status = 'completed'
  GROUP BY 1
)
SELECT
  month,
  revenue,
  SUM(revenue) OVER (ORDER BY month) AS running_revenue
FROM monthly
ORDER BY month;

6) Churn Flagged Users

SELECT
  user_id,
  start_date,
  end_date,
  churn_flag,
  CASE
    WHEN churn_flag = 1 THEN 'Churned'
    ELSE 'Active'
  END AS churn_status
FROM subscriptions;
Practice rule: If your query works but the result shape does not match the business question, the logic is probably still wrong.
Advanced Answers

Advanced Challenges Answer Key

These solved queries target higher-difficulty interview and job tasks for joins and window logic.

Challenge 1: Top 2 Customers Per Country by Revenue

WITH customer_revenue AS (
  SELECT
    u.country,
    o.user_id,
    SUM(o.quantity * o.unit_price - o.discount_amount) AS revenue
  FROM orders o
  JOIN users u
    ON o.user_id = u.user_id
  WHERE o.order_status = 'completed'
  GROUP BY 1, 2
)
SELECT *
FROM (
  SELECT
    country,
    user_id,
    revenue,
    ROW_NUMBER() OVER (PARTITION BY country ORDER BY revenue DESC) AS rn
  FROM customer_revenue
) ranked
WHERE rn <= 2
ORDER BY country, rn;

Challenge 2: Month over Month Growth by Country

WITH monthly_country AS (
  SELECT
    DATE_TRUNC('month', o.order_date) AS month,
    u.country,
    SUM(o.quantity * o.unit_price - o.discount_amount) AS revenue
  FROM orders o
  JOIN users u ON o.user_id = u.user_id
  WHERE o.order_status = 'completed'
  GROUP BY 1, 2
)
SELECT
  month,
  country,
  revenue,
  revenue - LAG(revenue) OVER (PARTITION BY country ORDER BY month) AS mom_change
FROM monthly_country
ORDER BY country, month;

Challenge 3: Users with No Purchases (Anti Join)

SELECT u.user_id, u.country, u.plan_type
FROM users u
LEFT JOIN orders o
  ON u.user_id = o.user_id
  AND o.order_status = 'completed'
WHERE o.order_id IS NULL
ORDER BY u.user_id;

Challenge 4: 7-Day Rolling Revenue

WITH daily_revenue AS (
  SELECT
    order_date::date AS dt,
    SUM(quantity * unit_price - discount_amount) AS revenue
  FROM orders
  WHERE order_status = 'completed'
  GROUP BY 1
)
SELECT
  dt,
  revenue,
  AVG(revenue) OVER (
    ORDER BY dt
    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
  ) AS rolling_7_day_avg
FROM daily_revenue
ORDER BY dt;

Sample Finance Query: Revenue vs Target

WITH invoice_rev AS (
  SELECT
    DATE_TRUNC('month', i.invoice_date) AS month,
    c.country,
    SUM(i.invoice_amount) AS invoiced_revenue
  FROM invoices i
  JOIN customers c ON i.customer_id = c.customer_id
  GROUP BY 1, 2
)
SELECT
  r.month,
  r.country,
  r.invoiced_revenue,
  b.revenue_target,
  r.invoiced_revenue - b.revenue_target AS variance_to_target
FROM invoice_rev r
LEFT JOIN monthly_budget b
  ON r.month = b.month
  AND r.country = b.country
ORDER BY r.month, r.country;
Finance Answers

Finance Answer Key (Solved Queries + Output Expectations)

Use this after solving the finance practice questions. Compare both query logic and result shape.

1) Overdue Invoices by Customer

SELECT
  c.customer_id,
  c.customer_name,
  c.country,
  i.invoice_id,
  i.invoice_date,
  i.due_date,
  i.invoice_amount,
  i.tax_amount,
  (i.invoice_amount + i.tax_amount) AS gross_due
FROM invoices i
JOIN customers c
  ON i.customer_id = c.customer_id
WHERE i.status = 'Overdue'
ORDER BY i.due_date;

Expected output: One row per overdue invoice with customer context and gross_due. Use it for collection follow-up queues.

2) Countries Below Monthly Revenue Target

WITH invoice_rev AS (
  SELECT
    DATE_TRUNC('month', i.invoice_date) AS month,
    c.country,
    SUM(i.invoice_amount) AS invoiced_revenue
  FROM invoices i
  JOIN customers c
    ON i.customer_id = c.customer_id
  GROUP BY 1, 2
)
SELECT
  r.month,
  r.country,
  r.invoiced_revenue,
  b.revenue_target,
  r.invoiced_revenue - b.revenue_target AS variance_to_target
FROM invoice_rev r
JOIN monthly_budget b
  ON r.month = b.month
  AND r.country = b.country
WHERE r.invoiced_revenue < b.revenue_target
ORDER BY r.month, variance_to_target;

Expected output: Only under-target country-month rows with negative variance_to_target values.

3) Collection Rate by Month and Country

WITH invoiced AS (
  SELECT
    DATE_TRUNC('month', i.invoice_date) AS month,
    c.country,
    SUM(i.invoice_amount + i.tax_amount) AS total_invoiced
  FROM invoices i
  JOIN customers c
    ON i.customer_id = c.customer_id
  GROUP BY 1, 2
),
collected AS (
  SELECT
    DATE_TRUNC('month', p.payment_date) AS month,
    c.country,
    SUM(p.payment_amount) AS total_collected
  FROM payments p
  JOIN invoices i
    ON p.invoice_id = i.invoice_id
  JOIN customers c
    ON i.customer_id = c.customer_id
  WHERE p.payment_status = 'Settled'
  GROUP BY 1, 2
)
SELECT
  i.month,
  i.country,
  i.total_invoiced,
  COALESCE(c.total_collected, 0) AS total_collected,
  CASE
    WHEN i.total_invoiced = 0 THEN 0
    ELSE ROUND(COALESCE(c.total_collected, 0) / i.total_invoiced::numeric, 4)
  END AS collection_rate
FROM invoiced i
LEFT JOIN collected c
  ON i.month = c.month
  AND i.country = c.country
ORDER BY i.month, i.country;

Expected output: One row per month-country with invoiced amount, collected amount, and collection_rate between 0 and 1.

4) Top Accounts by Paid Amount

WITH paid_amount AS (
  SELECT
    c.customer_id,
    c.customer_name,
    c.segment,
    SUM(p.payment_amount) AS paid_total
  FROM payments p
  JOIN invoices i
    ON p.invoice_id = i.invoice_id
  JOIN customers c
    ON i.customer_id = c.customer_id
  WHERE p.payment_status = 'Settled'
  GROUP BY 1, 2, 3
)
SELECT *
FROM (
  SELECT
    customer_id,
    customer_name,
    segment,
    paid_total,
    DENSE_RANK() OVER (ORDER BY paid_total DESC) AS paid_rank
  FROM paid_amount
) r
WHERE paid_rank <= 5
ORDER BY paid_rank, customer_name;

Expected output: Top customers by settled payments with ranks for account-prioritization and sales planning.

Beginner Level: SQL Fundamentals

Start with what SQL is, how relational databases work, and the minimum query patterns every analyst must know.

SQL Fundamentals

What is SQL?

SQL stands for Structured Query Language. It is used to ask questions of relational databases: retrieve, filter, group, join, and analyze structured data.

ConceptExplanationAnalyst Example
TableRows and columns stored togetherorders table
RowOne recordone order
ColumnOne attributeorder_date
Primary keyUnique row identifieruser_id in users
Foreign keyReference to another table's keyorders.user_id links to users.user_id

How Relational Databases Work

  • Data is normalized into related tables.
  • Primary keys identify rows uniquely.
  • Foreign keys connect business entities.
  • Queries combine tables to create analysis-ready views.
Analyst mindset: SQL is not only about selecting rows. It is about modeling a business question into a precise query structure.

Basic Query Pattern

SELECT -> FROM -> WHERE -> GROUP BY -> HAVING -> WINDOW -> ORDER BY

Core Syntax, Explanation, and Example

Clause Syntax Explanation Real-World Example
SELECT SELECT column1, column2 Chooses the columns you want to see. SELECT user_id, country FROM users;
FROM FROM table_name Specifies the source table. FROM orders
WHERE WHERE condition Filters rows before grouping. WHERE order_status = 'completed'
DISTINCT SELECT DISTINCT column Removes duplicate values. SELECT DISTINCT country FROM users;
LIMIT LIMIT n Returns only the first n rows. SELECT * FROM orders LIMIT 10;
ORDER BY ORDER BY column ASC|DESC Sorts results. ORDER BY order_date DESC

Basic Query Examples

-- Find all active users from the United States
SELECT user_id, signup_date, plan_type
FROM users
WHERE country = 'United States' AND is_active = 1
ORDER BY signup_date DESC
LIMIT 20;

-- Get unique acquisition channels
SELECT DISTINCT acquisition_channel
FROM users
ORDER BY acquisition_channel;
Real-world use: These are the first queries analysts use when checking source data, validating dashboards, or creating quick extracts for stakeholders.

Intermediate Level: Core Data Analysis SQL

Filtering and Aggregation

WHERE vs HAVING

ClauseWhen It RunsUse Case
WHEREBefore groupingFilter raw rows
HAVINGAfter groupingFilter aggregates
 
-- WHERE filters rows first
SELECT country, COUNT(*)
FROM users
WHERE is_active = 1
GROUP BY country;

-- HAVING filters grouped results
SELECT country, COUNT(*) AS user_count
FROM users
GROUP BY country
HAVING COUNT(*) > 100;

Aggregation Example

-- Revenue by country for completed orders
SELECT
  u.country,
  COUNT(o.order_id) AS order_count,
  SUM(o.quantity * o.unit_price - o.discount_amount) AS net_revenue,
  AVG(o.quantity * o.unit_price - o.discount_amount) AS avg_order_value
FROM users u
JOIN orders o
  ON u.user_id = o.user_id
WHERE o.order_status = 'completed'
GROUP BY u.country
ORDER BY net_revenue DESC;

Joins combine tables into one analysis view. This is one of the most important analyst skills.

Join Type Syntax Meaning Business Example
INNER JOIN FROM A INNER JOIN B ON A.id = B.id Only matching rows from both tables. Users with at least one order.
LEFT JOIN FROM A LEFT JOIN B ON A.id = B.id All rows from left table plus matches from right. All users, even those with no orders.
RIGHT JOIN FROM A RIGHT JOIN B ON A.id = B.id All rows from right table plus matches from left. All orders, even if user record is missing.
FULL OUTER JOIN FROM A FULL OUTER JOIN B ON A.id = B.id All rows from both tables. Reconciliation between systems.
CROSS JOIN FROM A CROSS JOIN B Every combination of rows. Scenario modeling or calendar expansion.
SELF JOIN FROM A a JOIN A b ON ... Table joined to itself. Employee-manager hierarchies.

Join Examples

-- INNER JOIN: active customers with orders
SELECT u.user_id, u.country, o.order_id, o.order_date, o.amount
FROM users u
INNER JOIN orders o
  ON u.user_id = o.user_id;

-- LEFT JOIN: keep all users and show orders if they exist
SELECT u.user_id, u.country, o.order_id
FROM users u
LEFT JOIN orders o
  ON u.user_id = o.user_id;

-- SELF JOIN: find each user's referrer or manager relationship
SELECT e.user_id, e.referral_user_id, r.signup_date AS referral_signup_date
FROM users e
LEFT JOIN users r
  ON e.referral_user_id = r.user_id;

Join Output Walkthrough

Join Type What Happens to Rows Example Result Behavior
INNER JOIN Rows survive only when both tables match. User 6 disappears if they have no order.
LEFT JOIN All left rows survive, right side may be null. All users appear, even if order_id is null.
FULL OUTER JOIN All rows from both tables survive. Shows unmatched users and unmatched orders for reconciliation.
CROSS JOIN Every combination is created. All product-country combinations for scenario planning.
SELF JOIN A row is matched to another row in same table. Referral or manager relationships among users.

Business Join Example: Users to Orders

SELECT
  u.user_id,
  u.country,
  u.plan_type,
  o.order_id,
  o.order_date,
  o.quantity * o.unit_price - o.discount_amount AS net_revenue
FROM users u
LEFT JOIN orders o
  ON u.user_id = o.user_id
ORDER BY u.user_id, o.order_date;

Interpretation: each user appears once for each order. If a user has no order, the order columns stay NULL. This is how analysts build customer-level reporting tables.

Memory rule: INNER = intersection, LEFT = keep left table, RIGHT = keep right table, FULL = everything, CROSS = all combinations, SELF = same table relationship.

Window Functions: Deep Analyst Guide

Window functions are essential for ranking, running totals, moving averages, time comparisons, and customer behavior analysis. They compute across a set of rows without collapsing the result like GROUP BY does.

Window Function Structure

FUNCTION(...) OVER (
  PARTITION BY partition_column
  ORDER BY sort_column
  ROWS BETWEEN ...
)
Component Syntax Explanation Why Analysts Use It
PARTITION BY PARTITION BY customer_id Splits rows into groups for window calculations. Per customer, per region, per month metrics.
ORDER BY ORDER BY order_date Defines row sequence inside the window. Running totals, rank, prior/next row comparison.
ROWS BETWEEN ROWS BETWEEN 6 PRECEDING AND CURRENT ROW Defines row-based window frame. 7-day moving average, rolling sums.
RANGE BETWEEN RANGE BETWEEN INTERVAL '30' DAY PRECEDING AND CURRENT ROW Defines value-based frame. Time-window analytics when dates are continuous.

Ranking Functions

FunctionSyntaxMeaningExample Use Case
ROW_NUMBER()ROW_NUMBER() OVER (...)Unique sequence numberRemove duplicates or assign latest row
RANK()RANK() OVER (...)Equal values share rank and leave gapsTop revenue customers with ties
DENSE_RANK()DENSE_RANK() OVER (...)Equal values share rank without gapsProduct ranking by category
NTILE()NTILE(4) OVER (...)Splits rows into equal bucketsCustomer quartiles for segmentation

ROW_NUMBER Example: Remove Duplicates

WITH ranked_orders AS (
  SELECT
    o.*,
    ROW_NUMBER() OVER (
      PARTITION BY order_id
      ORDER BY order_date DESC
    ) AS rn
  FROM orders o
)
SELECT *
FROM ranked_orders
WHERE rn = 1;

This keeps the newest record per order_id. Analysts use this pattern constantly when cleaning duplicate transactions, users, or events.

RANK and DENSE_RANK Example: Top Customers

SELECT
  user_id,
  SUM(amount) AS total_spend,
  RANK() OVER (ORDER BY SUM(amount) DESC) AS spend_rank
FROM orders
GROUP BY user_id;

Running Totals Example

SELECT
  order_date,
  amount,
  SUM(amount) OVER (
    ORDER BY order_date
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS running_revenue
FROM orders;

Moving Average Example

SELECT
  order_date,
  amount,
  AVG(amount) OVER (
    ORDER BY order_date
    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
  ) AS moving_7_row_avg
FROM orders;

LAG and LEAD Example: Period Comparison

SELECT
  order_date,
  amount,
  LAG(amount) OVER (ORDER BY order_date) AS previous_amount,
  amount - LAG(amount) OVER (ORDER BY order_date) AS change_from_previous,
  LEAD(amount) OVER (ORDER BY order_date) AS next_amount
FROM orders;

FIRST_VALUE and LAST_VALUE Example

SELECT
  user_id,
  order_date,
  amount,
  FIRST_VALUE(amount) OVER (
    PARTITION BY user_id
    ORDER BY order_date
  ) AS first_order_amount,
  LAST_VALUE(amount) OVER (
    PARTITION BY user_id
    ORDER BY order_date
    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
  ) AS last_order_amount
FROM orders;
Important: LAST_VALUE often surprises people because its default frame may stop at the current row. Use an explicit frame when you need the true last row in the partition.

Recursive CTE Example

WITH RECURSIVE org_tree AS (
  SELECT employee_id, manager_id, employee_name, 0 AS level
  FROM employees
  WHERE manager_id IS NULL
  UNION ALL

)
SELECT * FROM org_tree;

Analytical Query Patterns

Cohort Analysis

Problem: Which signup cohort retains best over time?

Pattern: Group users by signup month and compare later activity.

Visual model: Signup month becomes the row label; each later activity month becomes a follow-up column so you can read retention like a matrix.
WITH first_purchase AS (
  SELECT user_id, MIN(order_date) AS first_order_date
  FROM orders
  GROUP BY user_id
)
SELECT
  DATE_TRUNC('month', first_order_date) AS cohort_month,
  COUNT(*) AS users
FROM first_purchase
GROUP BY 1;

Retention Analysis

Problem: How many users return in later periods?

Pattern: Compare cohort month to activity month.

Visual model: Each cohort is a starting row, and retention decays over time from left to right.
SELECT
  cohort_month,
  activity_month,
  COUNT(DISTINCT user_id) AS active_users
FROM user_activity
GROUP BY 1, 2;

Funnel Analysis

Problem: Where do users drop off in conversion flow?

Pattern: Count users at each event stage.

Visual model: Each step is a narrower bar: visit -> signup -> checkout -> purchase.
SELECT event_name, COUNT(DISTINCT user_id)
FROM events
WHERE event_name IN ('visit', 'signup', 'checkout', 'purchase')
GROUP BY event_name;

Session Analysis

Problem: How long do users stay and what do they do in sessions?

Pattern: Use session_id with timestamps to calculate duration.

Visual model: Session timeline shows event order; duration is the distance between the first and last event.
SELECT
  session_id,
  MIN(event_time) AS session_start,
  MAX(event_time) AS session_end,
  EXTRACT(EPOCH FROM MAX(event_time) - MIN(event_time)) AS session_seconds
FROM events
GROUP BY session_id;

Time-Series Analysis

Problem: What is trend, seasonality, or anomaly behavior over time?

Pattern: Aggregate by day/week/month, then compare with prior periods.

Visual model: Put time on the x-axis, metric on the y-axis, and inspect slope, spikes, and recurring cycles.
SELECT
  DATE_TRUNC('month', order_date) AS month,
  SUM(amount) AS revenue
FROM orders
GROUP BY 1
ORDER BY 1;

Data Modeling Concepts

Fact Table

Contains measurable events such as orders, transactions, or sessions.

Example: orders(order_id, user_id, product_id, amount, order_date)

Dimension Table

Contains descriptive attributes such as customer, product, or date metadata.

Example: users(user_id, country, plan_type, signup_date)

Star Schema

One central fact table connected to several dimensions. Common in analytics warehouses.

Snowflake Schema

Dimension tables are normalized into sub-dimensions. Saves duplication but can add join complexity.

Analyst takeaway: Good data modeling makes reporting faster, simpler, and less error-prone.

Optimization Examples

-- Better: filter the smallest set first
WITH completed_orders AS (
  SELECT order_id, user_id, amount
  FROM orders
  WHERE order_status = 'completed'
)
SELECT u.country, SUM(c.amount)
FROM completed_orders c
JOIN users u ON c.user_id = u.user_id
GROUP BY u.country;
Rule of thumb: If a query is slow, ask whether you can reduce rows earlier, join less data, or use pre-aggregated tables.

Real-World Case Studies

1) Sales Dashboard Queries

Problem: Track revenue, orders, average order value, and growth by month and country.

SQL solution: Join users and orders, group by month and country, and compute net revenue.

SELECT
  DATE_TRUNC('month', o.order_date) AS month,
  u.country,
  COUNT(*) AS orders,
  SUM(o.quantity * o.unit_price - o.discount_amount) AS revenue
FROM orders o
JOIN users u ON o.user_id = u.user_id
GROUP BY 1, 2
ORDER BY 1, 2;

Explanation: This query feeds dashboards for executives and analysts.

2) Customer Segmentation

Problem: Split customers into high, medium, and low spend groups.

SQL solution: Aggregate spend per user, then bucket with CASE or NTILE.

WITH spend AS (
  SELECT user_id, SUM(amount) AS total_spend
  FROM orders
  GROUP BY user_id
)
SELECT
  user_id,
  total_spend,
  NTILE(4) OVER (ORDER BY total_spend DESC) AS spend_quartile
FROM spend;

Explanation: Useful for CRM, targeting, and lifecycle marketing.

3) Revenue Trend Analysis

Problem: Identify trend and month-over-month change.

SQL solution: Aggregate by month and use LAG.

WITH monthly AS (
  SELECT DATE_TRUNC('month', order_date) AS month, SUM(amount) AS revenue
  FROM orders
  GROUP BY 1
)
SELECT
  month,
  revenue,
  revenue - LAG(revenue) OVER (ORDER BY month) AS mom_change
FROM monthly;

Explanation: Standard pattern for revenue reporting and business review decks.

4) Churn Detection

Problem: Identify users who stopped engaging or cancelled their subscription.

SQL solution: Use subscriptions and activity gaps, then flag churned users.

SELECT
  user_id,
  end_date,
  CASE WHEN end_date IS NOT NULL THEN 1 ELSE 0 END AS churn_flag
FROM subscriptions;

Explanation: Used to build churn alerts and retention pipelines.

5) Top-N Analysis

Problem: Show top 10 customers, products, or channels.

SQL solution: Rank by metric and filter rank <= N.

WITH customer_spend AS (
  SELECT user_id, SUM(amount) AS total_spend
  FROM orders
  GROUP BY user_id
)
SELECT *
FROM (
  SELECT user_id, total_spend, RANK() OVER (ORDER BY total_spend DESC) AS rnk
  FROM customer_spend
) x
WHERE rnk <= 10;

Explanation: Common in sales, marketing, and executive reporting.

Memory Tricks and Mental Models

JOIN Visual Memory

  • INNER = overlap only
  • LEFT = keep left side
  • RIGHT = keep right side
  • FULL = union of both sides
  • CROSS = every combination
  • SELF = same table compared to itself

Window Function Pattern

Think: group inside the window, sort inside the group, compute across rows without collapsing them.

  • PARTITION BY = which group
  • ORDER BY = sequence inside the group
  • FRAME = how many rows or how much range

Query Building Order

SELECT -> FROM -> JOIN -> WHERE -> GROUP BY -> HAVING -> WINDOW -> ORDER BY

Mastery-Level Thinking

How Analysts Think Using SQL

Analysts think in business entities, metrics, grain, and time. They ask: What is the level of detail? What is the business question? Which table is the truth source? Where can duplication happen?

Common Mistakes in Real Jobs

  • Joining at the wrong grain.
  • Using GROUP BY too early or too late.
  • Filtering after aggregation when the question needs raw-row filtering.
  • Confusing counts of events with counts of users.
  • Forgetting that one user can create many rows.

Patterns Used by Strong Teams

  • CTEs to make logic readable and testable.
  • Window functions for rankings and trends.
  • Summary tables for dashboard performance.
  • Reusable date spine and cohort models.

FAANG-Level Habit

Break the business problem into measurable steps: define grain, find the base set, aggregate carefully, validate counts, and sanity-check outputs against intuition.
