For beginners

How to Use This Guide

If you are new, read the sections in order and build one small dashboard while you learn. Do not try to memorize Power BI first. Build a report with a sample sales table, then come back and connect the steps to what you created.

Beginner rule: every concept in Power BI becomes easier if you ask three questions: What is the data shape, what calculation is needed, and what business decision will this visual help with?
Recommended practice datasets in this workspace:
  • Sales: use order, customer, product, region, and finance tables.
  • Users: use customer signup, activity, and segmentation fields.
  • Finance: use invoice, payment, and budget tables for collections and variance analysis.

1. Power BI Fundamentals

What Power BI Is

Power BI is Microsoft's business intelligence platform for connecting data, cleaning it, modeling it, and building interactive dashboards. It is used when a team needs reliable reporting that refreshes and can be shared across the business.

ComponentWhat it doesBusiness use case
Power BI DesktopBuild reports, models, and DAX measuresAnalyst creates the first version of a dashboard
Power BI ServicePublish, refresh, share, and govern reportsTeam consumes a live sales dashboard
Power BI MobileView reports on phones and tabletsManagers monitor KPIs on the move

Interface Overview

  • Report view: build visuals and interactions.
  • Data view: inspect loaded tables and columns.
  • Model view: create relationships between tables.
  • Fields pane: where tables, columns, and measures live.
  • Visualizations pane: choose chart types, cards, maps, and filters.
Common beginner mistake: trying to build charts before understanding relationships. In Power BI, a bad model creates confusing dashboards later.

Importing Data

Typical starting sources are Excel, CSV, SQL databases, SharePoint folders, and cloud sources.

SourceBest useExample
ExcelSmall team files and ad-hoc inputsMonthly budget tracker
CSVSimple flat files and exportsSales transactions export
SQL databaseOperational data and large tablesOrders, invoices, customers
Business use case: a sales analyst may pull orders from SQL, customers from CRM export, and budgets from Excel. Power BI combines them into one model for reporting.

2. Data Transformation in Power Query

Power Query is where clean data starts. Think of it as the place where raw tables become analysis-ready tables before DAX ever runs.

Cleaning Data

  • Remove duplicates so each row means one record.
  • Handle missing values by replacing, imputing, or filtering.
  • Trim spaces and fix case when text fields are inconsistent.
  • Change data types early so dates and numbers behave correctly.

Transformations You Will Use Often

  • Split columns like full name into first and last name.
  • Merge columns for a display label or key.
  • Filter rows to keep only relevant periods or regions.
  • Add custom columns for margin, status, or buckets.

M Language Basics

M is the language behind Power Query. You do not need to master it on day one, but you should understand what a query script looks like. 

let
    Source = Csv.Document(File.Contents("sales.csv"), [Delimiter=",", Encoding=65001, QuoteStyle=QuoteStyle.Csv]),
    PromotedHeaders = Table.PromoteHeaders(Source, [PromoteAllScalars=true]),
    ChangedType = Table.TransformColumnTypes(PromotedHeaders, {{"Order Date", type date}, {"Sales", type number}}),
    RemovedDuplicates = Table.Distinct(ChangedType, {"Order ID"}),
    CleanRegion = Table.TransformColumns(RemovedDuplicates, {{"Region", Text.Trim, type text}})
in
    CleanRegion
Business use case: a refreshable report often fails because dates are stored as text. Fixing the type in Power Query prevents broken time intelligence later.

3. Data Modeling

Strong models make DAX easier and dashboards faster. Weak models force you to write complicated formulas and often cause wrong totals.

Star Schema vs Flat Table

ModelWhat it looks likeWhen to use it
Star schemaOne fact table in the center, surrounded by dimensionsBest for reporting and analytics
Flat tableAll fields in one wide tableOnly okay for small demos or quick prototyping

Core Table Types

Table typeExamplesRole
Fact tableSales, orders, invoicesStores transactions and numeric values
Dimension tableCustomer, product, date, regionStores descriptive context for slicing and filtering

Relationships

ConceptMeaningExampleBusiness note
One-to-manyOne customer has many ordersCustomers[CustomerID] -> Sales[CustomerID]Most common and preferred
Many-to-manyMultiple rows on both sidesProducts and campaigns via a bridge tableUse carefully and only when needed
Filter directionHow filters flow through relationshipsUsually single direction from dimension to factReduces ambiguity and improves performance
Real-world dataset example: use Sales fact with Date, Customer, Product, and Region dimensions. A manager can then slice sales by month, category, territory, and customer segment without changing the model.

Simple Star Schema

4. DAX Deep Dive

DAX is the language that turns a Power BI model into a real analytical engine. If Power Query cleans data, DAX turns that data into business answers.

4.1 Basic DAX

Calculated Columns vs Measures

TypeWhen it runsBest forBusiness use case
Calculated columnAt refresh time, row by rowStatic attributes and row labelsCustomer tier, age bucket, profit flag
MeasureAt query time, based on filter contextAggregations and dynamic calculationsSales, margin, YoY growth, KPI cards

Essential Functions

FunctionSyntaxWhat it does
SUMSUM(Sales[Amount])Adds numeric values
AVERAGEAVERAGE(Sales[Amount])Returns mean value
COUNT / COUNTACOUNT(Table[Column])Counts numeric or nonblank values
DIVIDEDIVIDE(numerator, denominator, 0)Safer division with blank handling
Business use case: a sales KPI card should usually be a measure, because the number must change when the user selects a different date, region, or product category. 
Sales Amount = SUM(Sales[SalesAmount])
Average Order Value = DIVIDE([Sales Amount], DISTINCTCOUNT(Sales[OrderID]), 0)
Order Count = COUNT(Sales[OrderID])

4.2 Intermediate DAX

CALCULATE

Syntax: CALCULATE(expression, filter1, filter2, ...)

What it does: changes filter context so you can answer "sales for one region", "sales for last year", or "sales after removing one slicer".

Business use case: monthly sales for the West region only.

West Sales = CALCULATE([Sales Amount], 'Region'[RegionName] = "West")

FILTER, ALL, ALLEXCEPT, VALUES

  • FILTER: returns rows that meet a condition.
  • ALL: removes filters so you can compare against a full total.
  • ALLEXCEPT: clears all filters except selected columns.
  • VALUES: returns the current distinct values in context.
Percent of Total =
DIVIDE(
    [Sales Amount],
    CALCULATE([Sales Amount], ALL(Sales)),
    0
)

4.3 Context Understanding

ConceptExplanationSimple exampleWhy it matters
Row contextThe formula is evaluated one row at a timeCalculated column on Sales tableNeeded for row-level logic
Filter contextThe current filters from slicers, visuals, and CALCULATESales for a selected monthControls what the measure sees
Context transitionCALCULATE converts row context into filter contextUsing a row value inside an iteratorCritical for advanced measures
Beginner trap: many wrong DAX results happen because the analyst confuses row context with filter context. When totals look wrong, inspect the context first.

4.4 Advanced DAX

Time Intelligence

FunctionSyntaxBusiness use case
TOTALYTDTOTALYTD([Sales Amount], 'Date'[Date])Year-to-date revenue
SAMEPERIODLASTYEARCALCULATE([Sales Amount], SAMEPERIODLASTYEAR('Date'[Date]))Compare this year to last year
DATEADDCALCULATE([Sales Amount], DATEADD('Date'[Date], -1, MONTH))Previous period comparison
YoY Growth % =
DIVIDE(
    [Sales Amount] - CALCULATE([Sales Amount], SAMEPERIODLASTYEAR('Date'[Date])),
    CALCULATE([Sales Amount], SAMEPERIODLASTYEAR('Date'[Date])),
    0
)

Running Totals, Ranking, Share

Running Sales =
CALCULATE(
    [Sales Amount],
    FILTER(
        ALL('Date'[Date]),
        'Date'[Date] <= MAX('Date'[Date])
    )
)

Rank by Sales =
RANKX(ALL(Customer[CustomerName]), [Sales Amount], , DESC, Dense)

Business use case: identify top customers, rolling revenue, and monthly trends for management reviews.

6. Advanced Power BI Features

Row-Level Security

RLS restricts what a user can see. It is used when regional managers should only see their own region or when finance should see only approved data. 

-- Example role filter on Region table
Region[RegionName] = "West"

Bookmarks, Buttons, and Dynamic Titles

Use bookmarks to create guided storytelling and buttons to move between pages or states.

Dynamic Title =
"Sales Dashboard - " & SELECTEDVALUE('Region'[RegionName], "All Regions")

Parameters

Parameters are useful for what-if analysis, dynamic thresholds, and flexible calculations. A classic example is margin target or discount scenario testing.

Business use case: a sales leader can test how revenue changes if price increases by 2 percent or if the discount threshold changes.

7. Performance Optimization

Reduce Model Size

  • Remove unused columns and rows.
  • Prefer numeric keys over long text keys.
  • Use star schema instead of one huge table.
  • Disable auto date/time when it creates unnecessary tables.

Optimize DAX

  • Avoid unnecessary iterators when a simple aggregation works.
  • Use measures instead of many calculated columns when possible.
  • Keep filters simple and readable.
  • Test performance with real slicers and page interactions.
AreaBad patternBetter pattern
ModelingWide flat table with repeated attributesFact table plus clean dimensions
RelationshipsMany bidirectional relationships everywhereSingle direction unless there is a strong reason
DAXNested logic without testing contextBreak the calculation into smaller measures

8. Real-World Business Projects

Use these as portfolio projects or internal case studies. Each one should be built with a clear model, a small set of strong measures, and a simple story.

Sales Dashboard

Problem: track revenue, profit, top products, and regional trends.

Model: FactSales, DimDate, DimCustomer, DimProduct, DimRegion.

Measures:

Sales Amount = SUM(Sales[SalesAmount])
Gross Profit = SUM(Sales[SalesAmount]) - SUM(Sales[Cost])
Gross Margin % = DIVIDE([Gross Profit], [Sales Amount], 0)

Dashboard: KPI cards, monthly trend, top products, regional map, and drill-through to product detail.

Financial Dashboard

Problem: monitor invoicing, collections, overdue balances, and budget variance.

Model: FactInvoices, FactPayments, DimCustomer, DimDate, DimBudget.

Measures:

Invoiced Revenue = SUM(Invoices[InvoiceAmount])
Collected Revenue = SUM(Payments[PaymentAmount])
Collection Rate = DIVIDE([Collected Revenue], [Invoiced Revenue], 0)

Dashboard: invoice status cards, aging buckets, monthly collection trend, target variance by country.

Customer Analytics Dashboard

Problem: understand acquisition, retention, churn, and segment value.

Model: FactOrders, FactActivity, DimCustomer, DimDate, DimSegment.

Measures:

Active Customers = DISTINCTCOUNT(Sales[CustomerID])
Repeat Purchase Rate =
DIVIDE([Customers with 2+ Orders], [Active Customers], 0)

Dashboard: cohort view, retention trend, segment summary, churn risk indicators.

12. Interview Preparation

Common Power BI Questions

  • What is the difference between a calculated column and a measure?
  • Explain row context and filter context.
  • How do you design a star schema?
  • When would you use RLS?
  • Why is CALCULATE so important?

Practical Scenarios

  • Revenue dropped 15 percent this month. How would you investigate it?
  • Users complain the dashboard is slow. What would you check first?
  • Sales totals are wrong by region. What model issues could cause that?
Interview answer style: describe the business goal first, then the model, then the measure logic, and only then the final visual. That order sounds professional and is easy to follow.
Mastery level

13. How BI Developers Think

Senior BI developers think in terms of trust, performance, clarity, and decision impact. They do not build dashboards just because a chart is possible. They build them because the business needs a reliable answer to a real question.

Before building

Ask what decision the report supports, who will use it, how often it refreshes, and what action follows from each KPI.

While building

Clean the model, keep measures readable, use consistent naming, and test every visual against the source data.

After building

Check refresh, security, page speed, and whether the report still answers the original business problem.

This guide is designed to work as training material for beginners while still staying close to the standards used in real BI teams.
